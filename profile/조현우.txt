【LLM 기반 고객 응대 챗봇 개발 프로젝트】
기간: 2024.01 ~ 2024.06 (6개월)
역할: AI 리드 엔지니어

프로젝트 개요:
고객 문의를 자동으로 처리하여 상담원의 업무 부담을 줄이고 응대 속도를 향상시키는 AI 챗봇을 개발했습니다. GPT 기반 모델을 회사 도메인에 맞게 파인튜닝하여 높은 정확도를 달성했습니다.

담당 업무:
- 고객 문의 데이터 수집 및 전처리
- 프롬프트 엔지니어링 및 Few-shot Learning 적용
- LoRA를 활용한 효율적인 파인튜닝
- RAG(Retrieval-Augmented Generation) 시스템 구축

기술적 도전과 해결:
LLM의 Hallucination 문제를 해결하기 위해 RAG 아키텍처를 도입했습니다. Vector DB에 회사의 FAQ와 상품 정보를 저장하고, 관련 정보를 검색하여 LLM에 제공함으로써 답변의 정확도를 크게 향상시켰습니다. 또한 답변 생성 후 Fact-checking 로직을 추가하여 잘못된 정보 제공을 방지했습니다.

성과:
- 고객 문의 자동 처리율 60% 달성
- 평균 응대 시간 5분에서 30초로 단축
- 고객 만족도 85% 유지
- 상담원 업무 부담 40% 감소

8. 팀워크 경험
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

이전 프로젝트에서 데이터 사이언티스트와 백엔드 개발자 간의 협업 문제로 모델 배포가 지연되는 일이 많았습니다. 데이터 사이언티스트는 Jupyter Notebook에서 잘 작동하는 모델을 만들었지만, 프로덕션 환경으로 전환하는 과정에서 많은 어려움이 있었습니다.

이를 해결하기 위해 MLOps 파이프라인을 구축했습니다. 모델 개발 단계부터 배포를 고려하여 코드를 작성하도록 가이드라인을 만들었고, Docker를 활용하여 개발 환경과 프로덕션 환경을 일치시켰습니다.

또한 정기적인 "ML Clinic" 세션을 운영하여 데이터 사이언티스트와 엔지니어가 함께 모델 배포 이슈를 논의하고 해결하는 시간을 가졌습니다. 백엔드 개발자에게 ML 기본 개념을 교육하고, 데이터 사이언티스트에게 소프트웨어 엔지니어링 Best Practice를 공유했습니다.

결과적으로 모델 배포 시간이 평균 2주에서 2일로 단축되었고, 모델 성능 모니터링과 재학습 자동화로 지속적인 개선이 가능해졌습니다.

9. 문제해결 경험
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

프로덕션에 배포된 추천 모델의 성능이 시간이 지나면서 점점 떨어지는 문제가 발생했습니다. 초기에는 CTR 4%였던 것이 3개월 후 2%로 하락했고, 사용자 이탈이 증가하기 시작했습니다.

문제 분석:
모델 성능 지표와 데이터 분포를 분석한 결과, 사용자 행동 패턴이 변화했는데 모델은 과거 데이터로만 학습되어 새로운 패턴을 반영하지 못하고 있었습니다. 이를 Model Drift라고 하며, 특히 빠르게 변화하는 트렌드가 있는 서비스에서 흔히 발생하는 문제였습니다.

해결 과정:
1. 데이터 드리프트와 모델 드리프트를 감지하는 모니터링 시스템을 구축했습니다.
2. 주기적인 재학습 파이프라인을 자동화하여 매주 최신 데이터로 모델을 업데이트했습니다.
3. Online Learning 기법을 적용하여 실시간으로 사용자 피드백을 모델에 반영했습니다.
4. A/B 테스트를 통해 신규 모델과 기존 모델의 성능을 지속적으로 비교했습니다.

결과:
- CTR이 4.5%로 회복되고 지속적으로 유지됨
- 모델 성능 저하 조기 감지로 빠른 대응 가능
- 자동 재학습으로 모델 관리 공수 70% 감소
- 비즈니스 지표(매출, 사용자 참여도) 개선

이 경험을 통해 모델 배포 후 지속적인 모니터링과 업데이트의 중요성을 깊이 이해하게 되었습니다.

10. 리더십 경험
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

작년에 AI 조직의 MLOps 인프라 구축 프로젝트를 리드했습니다. 각 데이터 사이언티스트가 개별적으로 모델을 관리하면서 재현성 문제와 협업 어려움이 있었고, 이를 해결하기 위한 통합 플랫폼이 필요했습니다.

리더십 발휘 사례:

1. 요구사항 수집 및 우선순위 결정
