1. 자기소개
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

저는 사용자 손안에서 직접 체험되는 서비스를 만드는 것에 열정을 가진 모바일 개발자입니다. 대학에서 컴퓨터공학을 전공하며 Android 개발에 흥미를 느꼈고, 실제 수백만 사용자가 사용하는 앱을 만들며 큰 보람을 느끼고 있습니다.

Kotlin과 Swift를 활용한 네이티브 앱 개발부터 Flutter를 활용한 크로스 플랫폼 개발까지 다양한 경험을 보유하고 있습니다. 최신 아키텍처 패턴과 성능 최적화에 관심이 많으며, 사용자 경험을 최우선으로 생각합니다.

모바일 생태계의 변화를 빠르게 따라가며, 미래 지향적인 모바일 서비스를 만드는 개발자로 성장하고자 합니다.

2. 지원동기
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

귀사의 모바일 앱은 국내 다운로드 수 1위를 기록하며 모바일 서비스의 표준을 제시하고 있습니다. 특히 부드러운 사용자 경험과 뛰어난 성능으로 업계에서 벤치마킹 대상이 되고 있습니다. 이러한 최고 수준의 모바일 개발 환경에서 일하고 싶어 지원하게 되었습니다.

저는 4년간 Android와 iOS 네이티브 앱 개발 경험이 있으며, MVVM 아키텍처와 Jetpack Compose를 활용한 모던 Android 개발에 능숙합니다. 또한 앱 성능 최적화 경험이 있어 귀사의 대규모 모바일 서비스 개발에 기여할 수 있습니다.

귀사에서 수백만 사용자가 사용하는 서비스를 개발하며 모바일 기술의 최전선에서 일하고 싶습니다.

3. 입사 후 포부
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

입사 후 초기에는 회사의 앱 아키텍처와 코드 구조를 빠르게 파악하고, 기존 개발 프로세스와 코드 컨벤션을 익히는 데 집중하겠습니다. 작은 기능 개발부터 시작하여 앱의 전체적인 흐름을 이해하겠습니다.

6개월 후에는 주요 화면과 기능 개발을 담당하며, UI/UX 개선과 성능 최적화에도 기여하겠습니다. 1년 후에는 새로운 기능의 기술 설계에 참여하고, 아키텍처 개선에도 의견을 제시하고 싶습니다.

장기적으로는 모바일 파트의 테크 리드로서 기술 방향성을 제시하고, 주니어 개발자들을 육성하는 역할을 수행하고 싶습니다.

4. 성장과정
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

스마트폰이 처음 등장했을 때부터 모바일 기술에 관심이 많았습니다. 고등학교 때 처음으로 간단한 안드로이드 앱을 만들어보며 모바일 개발의 매력을 느꼈고, 이것이 진로를 결정하는 계기가 되었습니다.

대학에서 컴퓨터공학을 전공하며 모바일 프로그래밍 동아리에서 활동했습니다. 교내 버스 도착 알림 앱, 도서관 좌석 예약 앱 등을 개발하며 실전 경험을 쌓았고, Google Play에 출시하여 실제 사용자 피드백을 받는 경험을 했습니다.

졸업 후 모바일 전문 개발사에서 커리어를 시작하며 다양한 업종의 앱을 개발했습니다. 금융, 커머스, 소셜 미디어 등 여러 도메인의 앱 개발 경험을 쌓았고, 이후 서비스 회사로 이직하여 대규모 사용자를 가진 앱의 운영과 개선 경험을 쌓았습니다.

다양한 프로젝트를 통해 모바일 개발 전반의 역량을 키웠고, 앞으로도 변화하는 모바일 트렌드를 선도하는 개발자가 되겠습니다.

5. 성격의 장단점
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

저의 가장 큰 장점은 사용자 중심적 사고와 세심함입니다. 기능 개발 시 단순히 동작하는 것으로 만족하지 않고, 실제 사용자가 어떻게 사용할지 고민하며 UX를 개선합니다. 작은 애니메이션과 터치 피드백 하나까지 신경 쓰는 편입니다.

빠른 학습 능력도 강점입니다. 새로운 라이브러리나 프레임워크가 나오면 적극적으로 학습하고 프로젝트에 적용합니다. Jetpack Compose와 SwiftUI 같은 새로운 UI 프레임워크도 빠르게 습득했습니다.

단점으로는 UI 디테일에 집착하다 보니 개발 일정이 늦어질 때가 있습니다. 이를 개선하기 위해 MVP(Minimum Viable Product)를 먼저 만들고 점진적으로 개선하는 방식을 채택하고 있습니다.

기술 스택: Kotlin, Java, Swift, Jetpack Compose, SwiftUI, Coroutines, Flow, Combine, Room, Realm, Retrofit, Firebase

【소셜 커머스 앱 리뉴얼 프로젝트】
기간: 2023.02 ~ 2023.10 (9개월)
역할: Android 리드 개발자

프로젝트 개요:
레거시 코드로 작성된 쇼핑 앱을 최신 아키텍처와 Jetpack Compose로 전환하는 대규모 리뉴얼 프로젝트를 진행했습니다. 사용자 경험을 개선하고 개발 생산성을 높이는 것이 목표였습니다.

담당 업무:
- Clean Architecture 기반의 멀티 모듈 구조 설계
- Jetpack Compose를 활용한 UI 개발
- Hilt를 활용한 의존성 주입 시스템 구축
- 3명의 주니어 개발자 멘토링 및 코드 리뷰

기술적 도전과 해결:
기존 XML 기반 UI와 Jetpack Compose를 함께 사용해야 하는 과도기적 상황에서 AndroidView와 ComposeView를 활용한 상호 운용성을 구현했습니다. 또한 큰 이미지를 효율적으로 로딩하기 위해 Coil 라이브러리를 도입하고 메모리 최적화를 수행했습니다.

성과:
- 앱 크래시율 80% 감소
- 리스트 스크롤 성능 50% 향상 (60fps 유지)
- 새로운 기능 개발 시간 40% 단축
- Google Play 평점 3.8에서 4.5로 상승

8. 팀워크 경험
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

이전 프로젝트에서 iOS 개발자와 Android 개발자 간의 UI 구현 차이로 인해 플랫폼 간 일관성이 떨어지는 문제가 있었습니다. 같은 디자인을 받아도 구현 방식이 달라 최종 결과물에 차이가 있었고, 이로 인해 사용자 혼란과 QA 이슈가 발생했습니다.

이를 해결하기 위해 양 플랫폼 개발자가 함께 참여하는 디자인 리뷰 프로세스를 만들었습니다. Figma에서 공통 컴포넌트를 정의하고, 각 플랫폼에서 구현 가능한 방식으로 디자인을 조정했습니다.

또한 Storybook과 유사한 방식으로 각 플랫폼의 컴포넌트 카탈로그를 만들어 시각적으로 비교할 수 있도록 했습니다. 정기적으로 두 플랫폼의 화면을 나란히 놓고 차이점을 찾아 수정하는 세션을 가졌습니다.

결과적으로 플랫폼 간 UI 일관성이 크게 향상되었고, QA 이슈가 60% 감소했습니다. 사용자 피드백에서도 "iOS와 Android 경험이 동일하다"는 긍정적인 평가를 받았습니다.

9. 문제해결 경험
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

앱 출시 후 특정 기기에서만 크래시가 빈번하게 발생하는 문제가 있었습니다. Firebase Crashlytics에서는 같은 에러가 계속 보고되었지만, 재현이 어려워 원인을 찾기 힘든 상황이었습니다.

문제 분석:
크래시 로그를 상세히 분석한 결과, 특정 삼성 기기의 특정 OS 버전에서만 발생하는 것을 발견했습니다. 메모리 부족으로 인한 OutOfMemoryError였고, 큰 비트맵을 로딩할 때 발생했습니다.

해결 과정:
1. 이미지 로딩 시 기기의 화면 밀도와 메모리를 고려한 적절한 크기로 리사이징했습니다.
2. Glide에서 Coil로 이미지 로딩 라이브러리를 변경하여 메모리 효율성을 개선했습니다.
3. LruCache를 활용한 메모리 캐싱 전략을 수정하여 캐시 크기를 동적으로 조절했습니다.
4. LeakCanary를 활용하여 메모리 누수를 발견하고 수정했습니다.

결과:
- 해당 기기군의 크래시율 15%에서 0.5%로 감소
- 전체 앱 크래시율 70% 개선
- 앱 메모리 사용량 평균 30% 감소
- 저사양 기기 사용자의 만족도 향상

이 경험을 통해 다양한 기기 환경을 고려한 방어적 프로그래밍의 중요성을 배웠습니다.

10. 리더십 경험
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

작년에 모바일 개발 팀의 기술 부채 해소 프로젝트를 리드했습니다. 레거시 코드가 누적되어 새로운 기능 개발이 어려워지고, 버그 수정이 다른 버그를 유발하는 악순환이 반복되고 있었습니다.

리더십 발휘 사례:

1. 현황 파악 및 우선순위 결정
팀원들과 함께 코드베이스를 분석하여 기술 부채를 문서화했습니다. 영향도와 개선 난이도를 기준으로 우선순위를 정하고, 스프린트마다 일정 시간을 기술 부채 해소에 투자하기로 합의했습니다.

2. 점진적 마이그레이션 전략
한 번에 모든 것을 바꾸는 대신, 새로운 기능 개발 시 해당 모듈만 최신 아키텍처로 전환하는 전략을 채택했습니다. Strangler Fig 패턴을 적용하여 점진적으로 레거시를 대체했습니다.

3. 지식 공유
주 1회 "Clean Code Friday" 세션을 운영하여 Best Practice와 Refactoring 기법을 공유했습니다. 실제 코드를 함께 리뷰하며 개선 방법을 논의했습니다.

4. 자동화 도구 도입
SonarQube를 도입하여 코드 품질을 지속적으로 모니터링했습니다. 복잡도가 높은 코드, 중복 코드, 보안 취약점 등을 자동으로 탐지하고 개선했습니다.
